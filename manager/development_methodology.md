# 通用编程方法论（默认执行规则）

> 适用于任何新需求：先调研、再选型、后开发、再优化。除非明确授权跳过，否则不允许直接编码。

## 核心原则
1. 先调研，不先编码。
2. 优先复用成熟开源底座，避免重复造轮子。
3. 底座需满足开源且可商用许可（MIT / Apache-2.0 等）。
4. 架构模块清晰、边界明确、可持续优化。
5. 支持后续模块替换、流程优化和长期维护。
6. 优先构建最小可验证闭环（PoC），再扩展。

## 默认执行顺序（必须遵循）
1. 需求理解
2. 开源方案调研
3. 底座选择
4. 模块拆解
5. 模块优化分析
6. 流程优化分析
7. 实施路线规划
8. 编码与验证

## 阶段要求
### 1) 需求理解
- 系统解决的问题
- 目标用户
- 核心价值
- PoC 最小验证能力
- 范围拆分：必须实现 / 可延后 / 当前不做

### 2) 现成方案调研
重点检查：
- 开源与商用许可
- 社区活跃度
- 稳定维护
- 适配度
- 部署复杂度
- 二开难度

必须回答：
- 可复用主底座是否存在
- 可直接复用模块
- 需自研部分
- 可替换优化模块

### 3) 底座选择
必须说明：
- 为什么选它
- 解决了什么
- 缺失能力是什么
- 后续扩展点

### 4) 模块拆解
每个模块明确：职责 / 输入 / 输出 / 边界 / 当前实现。
分类：直接复用 / 二次开发 / 未来可替换。

### 5) 模块优化分析
检查：性能瓶颈、稳定性、可维护性、可替代性、可复用空间。

### 6) 流程优化分析
检查：冗余步骤、重复计算、低效数据流、自动化机会、复杂度控制。

### 7) 实施路线规划
- 第一阶段：PoC，跑通核心闭环
- 第二阶段：功能完善与稳定性提升
- 第三阶段：性能与扩展性优化

### 8) 编码与验证
编码前置条件：前七阶段文档齐备。
编码后必须回答：
- 验证了什么
- 还缺什么
- 下一轮重点是什么

## 编码前必备文档
1. 调研报告
2. 底座选择说明
3. 系统模块拆解
4. 模块优化分析
5. 流程优化分析
6. 实施路线图

> 若以上文档未完成：不允许进入开发编码。

## 开发约束
1. 不重复造已有成熟底座
2. 不忽视模块边界
3. 不随意扩大需求范围
4. 不未调研先编码
5. 不只写代码不做验证
6. 不忽视长期维护

## 自动化协作补充约束
1. 自动触发稳定可靠
2. 任务未完成前不得提前触发下一轮
3. 必须设置循环上限
4. 每轮建议数量有限制
5. 连续重复建议需终止循环
6. 自动化必须服务开发流程，不干扰开发
